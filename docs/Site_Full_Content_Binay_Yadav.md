## Overview

```
The creative production team at Lowe's India spent significant hours each week manually producing and quality-checking ad creatives in Figma. The process was repetitive and error-prone — the same formatting mistakes reappeared across batches, and each one required manual correction before work could progress.

I identified the problem, proposed the solution, and built it.
```

---

## The Challenge

```
Creative ad production at scale creates a specific category of friction: the work is largely systematic, but it is executed manually. Templates drift from brand guidelines. Sizes get misapplied. Typography rules break. Spacing becomes inconsistent. Each error is small in isolation, but across a high-volume production workflow the cumulative cost is significant — in time, in review cycles, and in the mental load placed on the designers doing the work.

The opportunity was not to redesign the creative output. It was to automate the parts of the process that should never require a designer's attention in the first place.
```

---

## Research & Discovery

```
The first step was mapping the existing production process end to end. I conducted internal interviews with the production team to understand where errors most frequently originated, which steps consumed the most time, and which tasks were rule-based enough to be codified reliably.

The research produced a clear picture: a significant portion of weekly production hours were spent on tasks that followed fixed, repeatable rules. Every one of those hours was an opportunity to automate — and a risk, because manual execution of rule-based tasks is where errors enter the system.
```

---

## What I Built

```
A custom Figma automation plugin for creative ad design production. The plugin automates the repetitive production steps and runs quality validation checks before export — catching errors at the source rather than at the review stage.
```

**The plugin handles:**

```
Automated layout application across ad size variants.

Brand compliance validation covering colour, typography, and spacing rules — flagging deviations before they reach the production queue.

Quality checks that catch recurring errors automatically, eliminating the category of mistake that had been requiring manual correction on every batch.

Batch processing of multiple creatives in a single run, replacing a workflow that previously required handling each variant individually.
```

---

## The Build Process

```
Designing the plugin required treating the production team as the user. The same research and empathy applied to any UX project — understanding what someone needs to do, where the current process fails them, and removing that friction — applied here. The user happened to be a colleague rather than a customer.

I designed the plugin's interface as a UX problem first: what does a designer need to see, confirm, and do at each stage? The plugin had to feel like a natural extension of the existing Figma workflow, not an interruption to it.

The technical implementation used the Figma Plugin API and JavaScript. I built iteratively, testing each function with the production team and refining based on real usage before extending to the next feature.
```

---

## The Result

```
45% reduction in production time. Recurring errors were eliminated entirely. The team redirected hours previously spent on repetitive production and error correction toward higher-value design work.

The plugin was adopted as part of the standard production workflow at Lowe's India, directly supporting the team's broader digital transformation goals.
```

---

```
Figma Plugin API · JavaScript · Figma · Internal workflow analysis · User interviews with production team · Iterative testing

## Overview

```
Backyard is Lowe's enterprise design system — the standardised framework that governs UI across Lowe's websites and digital applications. Version 4 (DS4) was a significant update, restructuring the system around modern Figma capabilities: Variables, advanced Auto Layout, and a rebuilt token architecture.

I contributed to DS4 as a core team member — working as the team's Figma SME across design tokens, component guidelines, and design-to-development alignment.
```

---

## The Challenge

```
Enterprise design systems exist to solve consistency at scale. When dozens of designers and engineers are building different parts of the same product, visual and functional drift is a constant risk. A decision made in one area diverges from another. A component built by one team is subtly different from the same component built by a different team. Over time, the product loses coherence and the cost of correcting it compounds.

DS4 was tasked with solving this at the foundation — not just documenting components, but establishing the token structure and naming conventions that all components are built on, so that design and code share a single source of truth.
```

---

## Design Tokens

```
My primary contribution was the design token layer — defining and structuring the colour, spacing, typography, and elevation tokens that DS4 components reference.

Tokens are what make a design system scalable. A change at the token level propagates to every component that references it — one update that would otherwise require editing dozens of components manually. The work involved decisions about token naming, hierarchy, and the relationship between primitive tokens — raw values — and semantic tokens — purpose-driven aliases that describe intent rather than appearance.

Token naming conventions were defined in direct collaboration with the engineering team. A token named one thing in Figma and another in code is a source of drift — exactly the kind of inconsistency a design system is supposed to prevent. Resolving that alignment at the specification stage, before components are built on top of it, was one of the most consequential decisions in the project.
```

---

## Component Guidelines

```
I authored usage documentation for the components I owned — covering variants, interactive states, do and don't examples, and the rationale behind each specification. Every guideline was tied to real user behaviour and Lowe's brand standards, reviewed with product and engineering before publication.

The measure of good design system documentation is whether a designer who has never seen a component can use it correctly on the first attempt — and whether an engineer can implement it without needing to ask what was intended. That was the bar for everything written.
```

---

## Design–Development Alignment

```
Collaborated directly with the engineering team throughout the DS4 build to ensure design decisions translated cleanly into code. This included naming convention alignment, token structure reviews, and clarification of component specifications at the point of implementation rather than after the fact.

The result was a handoff process that reduced back-and-forth significantly — and a system that design and engineering could both refer to as the same source of truth.
```

---

## The Result

```
30% reduction in design-to-development handoff time across the enterprise. Consistent user experiences across Lowe's websites and applications, with a shared design language between disciplines that reduced ambiguity on every downstream project that used the system.
```

---

## What Design System Work Requires

```
Contributing to a design system is a different kind of design work. There is no single user journey, no hero screen, no moment where you ship something a customer sees directly. The value is systemic — it shows in the consistency of every product that builds on top of it, and in the speed of every team that uses it.

The most important skill in this work is not visual design. It is precision in decision-making and clarity in documentation — making choices that will be correct across a wide range of future use cases, and writing them down well enough that people who were not in the room can apply them correctly.
```

---

```
Figma (Variables, Component Properties, Auto Layout) · Design token documentation · Cross-functional collaboration with engineering · Lowe's brand guidelines

## Overview

```
Shop by Job is a purchase feature for Lowe's Pro customers — professional contractors who buy materials in volume for active projects. I led the complete end-to-end design of this feature from initial user research through to launch, with full ownership at every stage: research, strategy, information architecture, wireframing, prototyping, and usability testing.
```

---

## The Problem

```
Pro customers arrive at Lowe's with a clear intent: they have a job to do and they need the materials for it. An electrical rewire. A bathroom renovation. A deck installation. But the existing Lowe's experience required them to translate that job into individual product category searches — find the cable, find the conduit, find the fittings, find the junction boxes — each one a separate task across separate sections of the platform.

This is not how professional contractors think or work. They think in jobs and projects, not in product departments. The experience had been designed for a consumer buying a single item, not a professional managing a project with real timelines and real consequences for incomplete orders.

The result was purchase journeys that required far more steps than necessary, with a direct impact on completion rates.
```

---

## Research

```
I conducted comprehensive user research with professional contractors — the primary audience for this feature. Research included user interviews, journey mapping, persona development, and synthesis of findings into a clear set of design requirements.

Research findings were presented to the product and operations teams and used as the foundation for the feature brief.
```

**What the research found:**

```
Contractors organise their purchasing around named jobs and project types, not product departments. The mental model the platform was built on — browse by department, find a product — was the wrong model entirely for this user.

The materials required for a given job type are largely predictable. Contractors working on the same type of project repeatedly knew this and expected the platform to reflect it.

Incomplete purchase trips have direct professional consequences. A missed item means a return visit to the store, a delayed project start, rescheduled labour, and the compounding cost of poor planning. For a contractor, an abandoned cart is not a minor inconvenience — it is a real operational failure.
```

---

## The Solution

```
A guided purchase flow that starts with the job type, not a product search.

The contractor selects the type of job they are working on. The system surfaces the materials typically needed for that job — grouped, pre-configured, and ready to adjust. The contractor modifies quantities as needed and adds the complete project order to cart in a single flow.

Instead of building a cart through a dozen individual product searches, the contractor makes three decisions: the job type, quantity adjustments, and checkout. The purchase journey is structured around how they actually think and work.
```

---

## Usability Testing

```
Three rounds of usability testing with Pro customers. Each round tested a specific stage of the flow: job-type selection and taxonomy in the first round, materials list review and editing in the second, and cart confirmation and order summary in the third. Design was iterated between rounds based on observed friction and direct participant feedback.

The job-type taxonomy was restructured after round one. The initial categories had been defined by the merchandising team around internal product logic — not the language contractors used naturally. The second round tested a revised taxonomy built from terms participants used unprompted in the first round. Completion and confidence improved measurably.
```

---

## The Result

```
20% increase in transaction completion rates compared to the previous individual-search flow. Pro customers completed purchase journeys they had previously abandoned. The feature launched on the Lowe's platform and delivered measurable improvement in Pro customer satisfaction.
```

---

## Reflection

```
The clearest lesson from this project was the difference between designing for a task and designing for a context. The task — buy materials — was not the hard part. The context — a professional under time pressure, managing a real project, with real consequences for getting the order wrong — was what shaped every design decision.

Getting the taxonomy right required listening more than designing. The categories that made sense from inside the organisation were not the categories that made sense to the person doing the work. The research made that gap visible, and the willingness of the product team to restructure the feature around contractor language rather than internal logic was what made the outcome possible.
```

## Overview

```
Guardian's Shields is a healthcare product. At Omnicom Media Group, I led the complete redesign of the product website — applying a human-centered design approach that combined user research, iterative usability testing, and accessibility compliance to deliver a 50% increase in site engagement.
```

---

## The Problem

```
The previous website was structured around the product rather than the person using it. It opened with features, technical specifications, and pricing — a logical sequence from a sales perspective, but entirely the wrong sequence for a healthcare audience.

Healthcare users arrive in a research mindset. They are trying to understand whether this product is relevant to their situation — evaluating the problem, not yet comparing solutions. The existing architecture assumed they had already decided they needed this category of product and were now in a buying decision. Users who needed the product most, those in early-stage research with no prior knowledge of the category, were leaving before they found the information that could help them.

The engagement data confirmed it. The site was losing people at the top of the journey, before the product had a chance to establish credibility.
```

---

## Research

```
I conducted user research with healthcare decision-makers — people actively evaluating health protection products at the time of the study. Sessions covered three areas: how users orient themselves when they arrive at an unfamiliar healthcare product page; what questions need to be answered before trust begins to form; and where the existing navigation caused them to exit without finding what they needed.

I developed detailed user personas from the research — specific enough to make real design decisions against. The personas captured not just who the users were, but the context of the decision they were navigating: the degree of prior knowledge, the level of anxiety, the criteria being used to evaluate whether a product was credible.

An accessibility audit ran in parallel with the user research, identifying contrast failures, focus order issues, and unclear labelling across the existing site.
```

---

## The Redesign

```
The information architecture was rebuilt around the mental model the research surfaced:

Understand the problem → Trust the product → Make a decision → Take action.

The previous structure opened with the product. The new structure opened with the user's context and concern — meeting them where they were, not where the company needed them to be.
```

**Specific changes made:**

```
Navigation reduced from 8 top-level items to 4, restructured around the questions users were actually asking rather than the categories the product team had defined internally.

Lead content rewritten to address the user's situation before describing the product — establishing relevance before establishing features.

Accessibility brought to WCAG compliance across the full site, covering contrast ratios, focus management, label clarity, and keyboard navigation.

Usability testing ran across multiple rounds with real users, with design iterations applied between each round based on what was observed.
```

---

## The Result

```
50% increase in site engagement. Time on site increased. Bounce rate decreased. Conversion to the contact and information pages improved significantly.

The redesign was recognised internally at Omnicom as a reference standard for healthcare client UX work.
```

---

## Reflection

```
Healthcare UX makes the stakes of information architecture unusually visible. A user who leaves a retail site without finding what they need has a minor inconvenience. A user who leaves a healthcare site without understanding whether a product is right for their situation may make a worse decision as a result.

Designing for that context required holding two things simultaneously: the emotional state of the user — uncertainty, a need for reassurance before information — and the practical requirements of what they needed to know. Getting the sequence right mattered more than getting the content right. The right information in the wrong order still loses people.
```

## Overview

```
Pro Central is a unified platform for Lowe's Pro customers — professional contractors who manage multiple active projects and rely on Lowe's as their primary materials partner. The platform consolidates project management, inventory tracking, invoicing, and order management into a single experience, designed around how contractors actually work rather than how a retail product is typically organised.
```

---

## The Problem

```
Pro customers are Lowe's highest-value segment. But the digital experience was not built for them — it was built for a consumer making a single purchase.

A contractor running an active project has a different set of needs entirely. They need to track materials ordered for each job, reorder stock when levels run low, generate invoices for their own clients based on what they've purchased through Lowe's, and verify store availability before committing to a project timeline. Before Pro Central, none of these needs were served by a connected experience. Each required a separate process — disconnected sections of the Lowe's platform, supplemented by spreadsheets and third-party tools.

The friction compounded across every project and every week. Every disconnected step was a point where information was re-entered, time was lost, or a mistake was made. For a contractor managing multiple active jobs simultaneously, this was not a minor inconvenience — it was a recurring operational cost.
```

---

## Research

```
I conducted user research with professional contractors using interviews and journey mapping. The central finding was consistent across every participant: contractors think and work in projects. Every purchasing decision is made in the context of a named job with a deadline. The platform's category-based structure was built around product types — the wrong model entirely for a user who arrives with a project in mind, not a search query.
```

**Additional research findings:**

```
Reordering is the most frequent action in the Pro customer workflow and involves the most unnecessary steps. Finding previous order details for a specific job required navigating multiple screens with no persistent project context.

Invoice management happened entirely outside the platform for every participant interviewed. The existing invoice functionality existed but was not discoverable, and was insufficient for professional use. Every contractor was maintaining invoices in a separate app or spreadsheet.

Inventory visibility was needed at the planning stage — when scheduling a project, before any purchase decisions are made — not only at the point of purchase. Contractors described the cost of arriving at a store to find key items out of stock: delayed project start, rescheduled labour, and the full logistical cost of a wasted trip.
```

---

## The Solution

```
Pro Central is organised around projects, not departments. The home view shows every active job — status, materials ordered, outstanding items, budget tracking, and invoice state — in a single screen. From that view, a contractor can reorder materials, generate an invoice, or check availability without switching tools or contexts.
```

**Core features:**

**Project Dashboard**

```
A unified view of every active job with the information a contractor needs to manage it — not a product catalogue with a project filter applied, but a genuine project management view that surfaces purchasing and invoicing in context.
```

**Reorder Flow**

```
Materials ordered for each project are stored and accessible from the project view. Current stock levels are shown inline. The most frequent action in the Pro workflow was given the most direct path.
```

**Integrated Invoicing**

```
Invoice generation rebuilt within the project context, with line items that auto-populate from order history. The goal was to make the tool contractors had been using externally redundant — by providing something that worked at least as well and required no context-switching.
```

**Inventory Visibility**

```
Real-time availability for saved project materials, accessible at the planning stage. Contractors can confirm stock levels before committing to a project schedule — eliminating the category of wasted trip that the research surfaced as a recurring frustration.
```

---

## Handoff

```
All components built on Backyard Design System 4. Handoff documentation included annotated Figma frames for all primary user flows, error and empty state screens, interaction specifications for key micro-interactions, and component-level notes mapping each element to its DS4 token.
```

---

## Reflection

```
Pro Central required designing for a user under genuine professional pressure. A contractor checking a reorder on a job site is not browsing — they have one task, limited time, and real consequences for delays. Every screen had to be evaluated against that context: is the most important action the most visible one? Is there anything here that requires a decision that could be eliminated entirely?

The research made the requirements clear. The challenge was holding those requirements consistently across a complex, multi-feature platform — ensuring that a decision made for the project dashboard did not introduce friction in the invoicing flow, and that the full system felt unified rather than assembled from separate parts.
```
